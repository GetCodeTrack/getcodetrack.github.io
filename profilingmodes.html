Profiling modes

Codetrack provides a wide range of options for profiling.
Also check out this post, where the difference is explained : The rocket tale (part I): Sampling vs Tracing

  • Sampling

    In this mode, the threads of your process are being suspended periodically and a snapshot of the call stack for every thread is taken. Use this mode to get a general idea about the performance of your code.
    Since snapshots are made periodically, this method does NOT capture every method call.
    This is the profiling mode that has the least impact on the performance of your process. The captured amount of data is relatively small and is processed very quickly.

  • Tracing

    Tracing means that EVERY call will be captured and all measurements will be 100% accurate. The more options you enable, the more information you will get, but the bigger your captured data will be and the slower your application will run. It will result in the same performance tree as the sampling mode, but here the timings will be exact and every call will be included. In tracing mode you also get a precise invocation count for every method.

    • Timeline

      Enable this option if you want to have the timeline in your analyze window. This will give you an incredibly detailed view on ALL of your method calls. Including their position in the call stack and their start and end time.
      If you don't enable this option, you will still get a timeline, but it will be a virtual one (more info about that here).

      If timeline is enabled, you can also choose to trace all method arguments. If you enable track objects, every object will have it's own unique id and can be identified.

    • Scope

      Scope enables you to select the code you want to be profiled. This can be done based on:

      • Assembly
      • File path
      • Method name
      • Type name
      • Thread name
      • Process name
      • Process path
